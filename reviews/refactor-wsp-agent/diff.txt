+# Documentación del Módulo: wsp-agent-v2 (WhatsApp Backend)
+
+**Última actualización:** 2026-06-23
+**Estado:** Activo / En Producción (Con mejoras pendientes)
+
+---
+
+## 1. Visión General
+
+`wsp-agent-v2` es el microservicio encargado de gestionar las conexiones de WhatsApp para múltiples inquilinos (Multi-Tenant WISP). Utiliza la librería **Baileys** (`@whiskeysockets/baileys`) bajo el capó para interactuar con el protocolo de WhatsApp Web.
+
+El objetivo principal de este módulo es aislar la complejidad, la inestabilidad inherente y el alto consumo de recursos que genera el protocolo de WhatsApp Web, separándolo del backend principal (`provision-agent-v2` / Next.js).
+
+### Responsabilidades Clave:
+*   Gestión concurrente de múltiples sesiones de WhatsApp (hasta 200 por defecto).
+*   Autenticación persistente en base de datos PostgreSQL (Supabase).
+*   Enrutamiento de mensajes entrantes mediante Webhooks hacia el agente principal.
+*   Envío de mensajes salientes a petición del panel o del agente de IA.
+*   Reconexión resiliente frente a caídas de red o desconexiones del servidor de Meta.
+*   Aislamiento de inquilinos (Tenant Isolation) basado en `wispId`.
+
+---
+
+## 2. Arquitectura de Archivos y Componentes
+
+| Archivo | Responsabilidad Principal |
+| :--- | :--- |
+| **`server.mjs`** | Punto de entrada. Levanta el servidor Express en el puerto 3001, monitorea el *Event Loop Lag* y la memoria, y orquesta el *Graceful Shutdown* global. |
+| **`session.manager.mjs`** | Motor principal de sesiones. Mantiene el `Map` en memoria de *sockets* activos, gestiona el *Idle Timeout* para liberar recursos de sesiones inactivas, implementa *Lazy Loading* (carga perezosa) y maneja la destrucción segura de sockets. |
+| **`event.handlers.mjs`** | Escucha y procesa eventos de Baileys (`messages.upsert`, `connection.update`). Implementa una doble capa de idempotencia (LRU Cache + Supabase) para evitar mensajes duplicados y reenvía los eventos al webhook del agente principal. |
+| **`database.auth.mjs`** | Adaptador de autenticación personalizado. Intercepta la lectura/escritura de credenciales de Baileys y las sincroniza con Supabase (`wsp_session_keys`). Utiliza una técnica de *Flush Híbrido* (Coalescing de 10s) para optimizar escrituras masivas a BD. |
+| **`reconnect.manager.mjs`** | Controlador de reconexiones. Implementa *Exponential Backoff* con *Jitter* y un *Circuit Breaker* básico. Identifica errores fatales (como cierres de sesión remotos) para detener la reconexión y destruir la sesión adecuadamente. |
+| **`outbound.service.mjs`** | Router de Express que expone la API REST para el panel o el agente. Permite enviar mensajes (`/send-message`), despertar sesiones (`/wake`), chequear salud (`/health`) y forzar desvinculaciones (`/logout-session`). |
+| **`socket.factory.mjs`** | Fábrica responsable de instanciar el socket de Baileys puro vinculándolo con el estado de autenticación correspondiente a su `wispId`. |
+
+---
+
+## 3. Flujos de Datos Principales
+
+### A. Flujo de Recepción de Mensajes (Inbound)
+1.  **WhatsApp / Meta** empuja un mensaje al socket de Baileys.
+2.  **`event.handlers.mjs`** captura el evento `messages.upsert`.
+3.  Se filtra: se ignoran mensajes del bot mismo, vacíos o gigantes (>4000 chars).
+4.  **Idempotencia:** Se consulta la caché LRU (Capa 1) y luego Supabase `wsp_processed_messages` (Capa 2) para verificar que el mensaje no sea un duplicado de un reinicio anterior.
+5.  Si es nuevo, se envía un `POST` al Webhook Principal (`MAIN_AGENT_WEBHOOK`) con un timeout estricto de 15 segundos y limitador de concurrencia por WISP.
+
+### B. Flujo de Envío de Mensajes (Outbound)
+1.  **El Agente (IA) o el Panel** llama al endpoint `POST /api/send-message`.
+2.  **`outbound.service.mjs`** intercepta la petición.
+3.  Solicita el socket a **`session.manager.mjs`** vía `lazyLoadSession`. Si el socket no existe en RAM pero sí en BD, lo levanta bajo demanda.
+4.  Se ejecuta `sock.sendMessage()` con un `Promise.race` (timeout de 15 segundos) para evitar colgar la respuesta HTTP si el socket está congelado.
+5.  Retorna el `messageId` original.
+
+### C. Flujo de Autenticación Híbrida (Base de Datos)
+1.  Al instanciar un socket, **`database.auth.mjs`** descarga las credenciales iniciales de Supabase de manera síncrona.
+2.  Cuando Baileys necesita actualizar llaves criptográficas (que ocurre docenas de veces por segundo en grupos o chats grandes), se almacenan en un `pendingKeys` en RAM.
+3.  Un temporizador (`flushTimer`) de 10 segundos recolecta todos esos cambios y hace un único `UPSERT` masivo a Supabase (`wsp_session_keys`). Si el contenedor muere repentinamente, `server.mjs` invoca un *Flush Forzado* durante el *Graceful Shutdown*.
+
+---
+
+## 4. Deuda Técnica y Backlog de Arquitectura (Known Issues)
+
+Este módulo es funcional y soporta multi-inquilinos, pero requiere las siguientes intervenciones críticas para alcanzar un grado 100% *Enterprise-Ready*:
+
+> [!WARNING]
+> **1. Race Condition en Idempotencia (`event.handlers.mjs`)**
+> Dos mensajes idénticos que lleguen al mismo milisegundo pueden atravesar el bloque `if` del LRU antes de que la BD o la RAM registre el primero.
+> *Solución sugerida:* Implementar lógica *Insert First* contra la BD o asegurar un *Unique Index* estricto a nivel Supabase en `wsp_processed_messages`.
+
+> [!IMPORTANT]
+> **2. Riesgo de Sesiones Zombis (`reconnect.manager.mjs` y `session.manager.mjs`)**
+> Existe una *Race Condition* entre la destrucción manual de una sesión (`destroySession`) y un temporizador pendiente (`setTimeout`) de reconexión de `reconnect.manager.mjs`. Si la reconexión se dispara después de destruir, la sesión revivirá en memoria sin estar controlada.
+> *Solución sugerida:* Implementar identificadores de "generación" de sesión (`instanceId = crypto.randomUUID()`) para validar que la reconexión pertenezca a la instancia actual válida.
+
+> [!TIP]
+> **3. Contadores de Webhooks (`event.handlers.mjs`)**
+> El control actual utiliza contadores manuales (`webhookCounters.get`) incrementados y decrementados. Esto es propenso a desbalances si hay excepciones atípicas.
+> *Solución sugerida:* Reemplazar con una cola de promesas controlada como `p-limit` o `p-queue`.
+
+> [!NOTE]
+> **4. Anti-loop de Inteligencia Artificial**
+> No hay un mecanismo nativo en `wsp-agent-v2` para detectar si el mensaje que llegó fue causado por un bot propio en otro dispositivo vinculado.
+> *Solución sugerida:* Guardar un array en memoria de los últimos `messageIds` despachados por IA para ignorarlos proactivamente.
+
+> [!WARNING]
+> **5. Tormenta de Reconexiones Masiva (Thundering Herd)**
+> En caso de una pérdida global de red en el VPS, las 200 sesiones intentarán reconectarse simultáneamente cuando vuelva el internet, lo que podría provocar bloqueos de IP por parte de Meta o saturar el CPU.
+> *Solución sugerida:* Implementar una cola global estricta para despachar reconexiones en lotes, además del Jitter actual.
+
